= Spark Boot User Guide
:toc: left
:toclevels: 3
:sectanchors:

Spark Boot is a JVM/Kotlin framework for Spark pipeline definition and execution.
It is built around four deliberately separate responsibilities:

[source,text]
----
Dagger = construction
Kotlin DSL = composition
HOCON DSL = declarative job description
Spark Runtime = execution
----

The central execution path is:

[source,text]
----
Kotlin DSL or HOCON DSL
  -> FlowDefinition / ExecutableFlow
  -> Flow / Node / Edge model
  -> Dagger-backed node factories
  -> SparkRuntime
  -> Spark execution
----

Spark Boot HOCON DSL targets SeaTunnel-style configuration shape compatibility.
It does not claim full Apache SeaTunnel engine, runtime, or connector compatibility.

== General Concepts

Spark Boot separates object construction from pipeline composition.
This is the most important rule to understand before using the Kotlin DSL.

[cols="1,3", options="header"]
|===
| Concept | Meaning

| Dagger component
| Owns framework construction: `SparkSession`, `SparkRuntime`, built-in node factories, and factory registries.

| Node factory
| Creates node instances. The Kotlin DSL asks Dagger for these factories, then calls `create()`.

| Flow
| A named directed acyclic graph of node instances and edges.

| Node id
| The id of a node inside one flow. It is not a Dagger bean name.

| Node config
| Mutable properties assigned on a node instance before the flow runs.

| `NodeRef<T>`
| A DSL handle for a registered node. Use it to chain or explicitly connect edges.

| Runtime
| Validates the assembled DAG and executes nodes in topological order.
|===

=== Flow IR

`FlowDefinition` is Spark Boot's runtime-independent graph model.
It is intentionally small:

* `name`
* `nodes`
* `edges`
* `config`

Tools should use `FlowDocument` as the serialized IR boundary.
`FlowDocument` wraps `FlowDefinition` with:

* `schemaVersion`, currently `spark-boot.flow/v1`
* `ui`, for layout and editor-only metadata

[source,json]
----
{
  "schemaVersion": "spark-boot.flow/v1",
  "flow": {
    "name": "paid-orders",
    "nodes": [
      {
        "id": "orders",
        "type": "ParquetSource",
        "config": {
          "path": "s3a://warehouse/orders"
        }
      }
    ],
    "edges": []
  },
  "ui": {
    "nodes": {
      "orders": {
        "x": 80,
        "y": 160
      }
    }
  }
}
----

`FlowDocument` is the recommended JSON format for UI builders and other tools.
HOCON remains the user-facing deployment/config DSL, and the Kotlin DSL remains
the developer-facing programmatic API.

Node descriptors describe available node types, roles, field schemas, defaults,
and enum options.
Validation should use `FlowDefinitionValidator` before a graph is assembled or
executed, so tools can show structured diagnostics without starting Spark.

=== Kotlin DSL Semantics

A Kotlin DSL source call has this shape:

[source,kotlin]
----
val orders = parquetSource("orders") {
    path = "data/orders"
}
----

The semantic contract is:

. `parquetSource` asks the current `SparkBootComponent` for a `ParquetSourceNodeFactory`.
. The factory creates a new `ParquetSourceNode` instance.
. The lambda is applied to that node instance.
. The node is registered in the current flow with id `orders`.
. A `NodeRef<ParquetSourceNode>` is returned.

Equivalent pseudo-code:

[source,kotlin]
----
val node = component.parquetSourceNodeFactory().create()
node.path = "data/orders"
val orders = builder.register("orders", node)
----

The string `"orders"` is a flow node id.
It is used for DAG registration, edge wiring, validation, and runtime lookup.
It is not a Dagger key, Dagger bean name, Spark table name, or output path.

The lambda is configuration only.
It should set node properties such as paths, SQL conditions, save modes, table names, JDBC options, and similar node settings.
It should not start Spark jobs, create new Spark sessions, or perform external side effects.
Spark work happens later when `SparkRuntime.run(flow)` executes the assembled graph.

Chained DSL calls add edges:

[source,kotlin]
----
parquetSource("orders") {
    path = "data/orders"
}
    .filterSql("paid-only") {
        condition = "status = 'PAID'"
    }
    .writeParquet("sink") {
        path = "output/paid-orders"
    }
----

This creates three nodes and two edges:

[source,text]
----
orders -> paid-only -> sink
----

Explicit references are useful when the graph is not a simple chain:

[source,kotlin]
----
val orders = parquetSource("orders") {
    path = "data/orders"
}

val paid = orders.filterSql("paid") {
    condition = "status = 'PAID'"
}

orders.filterSql("cancelled") {
    condition = "status = 'CANCELLED'"
}
----

Node ids must be unique inside a single flow.
The same id can be reused in another flow because ids are flow-local.

=== Execution Contract

At runtime, Spark Boot:

. validates that the flow contains nodes;
. validates that edges reference known node ids;
. validates the graph is acyclic;
. validates each node has at most one input in the current version;
. executes nodes in topological order;
. passes source nodes `Unit`;
. passes transform and sink nodes the upstream node output.

Dagger is not consulted by node id during execution.
The node instances already exist in the `ExecutableFlow`.
Runtime execution uses the flow's node map and edge list.

=== Driver Code And Executor Code

Spark Boot DSL code runs on the Spark driver.
This includes Kotlin DSL lambdas such as:

[source,kotlin]
----
parquetSource("orders") {
    path = "data/orders"
}
----

That lambda configures a node instance while the flow is being assembled.
It is not sent to Spark executors.
The same is true for built-in nodes such as `SqlFilterNode`, `SelectNode`,
`ParquetSourceNode`, `JdbcSourceNode`, `ParquetSinkNode`, and `IcebergSinkNode`:
they run driver-side code that builds Spark DataFrame, SQL, read, or write plans.
Spark then distributes Spark's own physical execution plan to executors.

This is why normal Spark Boot DSL usage does not usually hit Kotlin lambda
serialization problems.
It is not because Spark has special Kotlin serialization support.
Kotlin compiles to JVM bytecode, and Spark's normal closure serialization rules
still apply whenever user code enters executor-side Spark APIs.

Executor-side code starts when a user-provided node calls APIs such as:

* `Dataset.map`
* `Dataset.foreach`
* `Dataset.foreachPartition`
* `Dataset.rdd().map`
* `Dataset.javaRDD().map`
* `spark.udf().register` with a JVM lambda or function object

Avoid capturing application services, Dagger components, Spark sessions,
database clients, loggers with non-serializable state, or other driver-only
objects in those closures.

Unsafe example:

[source,kotlin]
----
class Formatter(private val prefix: String) {
    fun format(id: String): String = "$prefix$id"
}

class UnsafeSink(
    private val formatter: Formatter
) : SparkSinkNode<Dataset<Row>> {
    override val name: String = "unsafe-sink"

    override fun execute(input: Dataset<Row>, context: SparkExecutionContext) {
        input.javaRDD()
            .map { row -> formatter.format(row.getAs<String>("id")) }
            .count()
    }
}
----

`formatter` must be serialized with the Spark task closure.
If it is not serializable, or if it references a larger non-serializable object
graph, the job can fail with `Task not serializable` or
`NotSerializableException`.

Preferred patterns:

* Use built-in Spark SQL/DataFrame expressions: `filterSql`, `select`, and `sqlTransform`.
* Keep node lambdas as configuration only.
* If executor logic is unavoidable, capture only small serializable values.
* Put reusable executor functions in top-level classes or `object` methods.
* Broadcast large read-only data explicitly with `spark.sparkContext.broadcast`.
* Test executor closures with `local-cluster[...]` or a real cluster, not only `local[*]`.

Safe executor-side shape:

[source,kotlin]
----
class SafeSink(
    private val prefix: String
) : SparkSinkNode<Dataset<Row>> {
    override val name: String = "safe-sink"

    override fun execute(input: Dataset<Row>, context: SparkExecutionContext) {
        val serializablePrefix = prefix
        input.javaRDD()
            .map { row -> "$serializablePrefix${row.getAs<String>("id")}" }
            .count()
    }
}
----

`local[*]` runs executors in the same JVM process and can hide classpath and
deployment problems.
Spark's `local-cluster[...]` mode starts standalone worker and executor JVMs,
so it is closer to cluster behavior.
Spark Boot includes an opt-in local-cluster integration test for this:

[source,bash]
----
SPARK_HOME=/path/to/spark-4.0.1-bin-hadoop3 \
  ./gradlew :integration-tests:test \
  --tests org.openprojectx.spark.boot.integration.SparkBootLocalClusterIntegrationTest \
  --no-configuration-cache
----

The test is skipped when `SPARK_HOME`, `spark.home`, or `spark.test.home` is not
set to a Spark distribution.
Maven-resolved Spark jars alone are not enough for `local-cluster[...]` because
Spark's standalone launcher expects a distribution layout with a `jars`
directory.

Spark Boot currently targets:

* Java 17
* Kotlin 2.2.x
* Spark 4
* Scala 2.13 Spark artifacts
* Gradle Kotlin DSL
* Dagger 2
* Typesafe Config

Spark dependencies are managed with the Spark Platform Gradle plugin:

----

Spark-facing modules set:

----

    line.set("spark4")
}

== Module Responsibilities

=== core

The `core` module contains only portable Spark Boot model types.
It does not depend on Spark, Dagger, or HOCON.

Important types:

* `FlowDefinition`
* `NodeDefinition`
* `EdgeDefinition`
* `ExecutableFlow`
* `FlowNode`
* `NodeFactory`
* `ConfigNodeFactory`
* `NodeFactoryRegistry`
* `FlowAssembler`

Use `FlowDefinition` for declarative descriptions, such as HOCON.
Use `ExecutableFlow` when actual runtime node instances have already been created.

=== runtime-spark

The `runtime-spark` module contains Spark-specific execution contracts:

* `SparkExecutionContext`
* `SparkNode`
* `SparkSourceNode`
* `SparkTransformNode`
* `SparkSinkNode`
* `SparkRuntime`

`SparkRuntime` validates and executes a DAG.
The first implementation supports one input per transform or sink.
Nodes with multiple incoming edges fail with a clear error.

=== connectors

The `connectors` module provides built-in nodes:

* `ParquetSourceNode`
* `ParquetSinkNode`
* `JdbcSourceNode`
* `SqlFilterNode`
* `SelectNode`
* `SqlTransformNode`
* `IcebergSinkNode`
* `JdbcSinkNode`

It also provides Dagger-friendly programmatic factories and HOCON config factories for those nodes.

=== dagger

The `dagger` module wires Spark Boot construction:

* `SparkBootComponent`
* `SparkModule`
* `RuntimeModule`
* `BuiltinConnectorModule`

Dagger constructs Spark sessions, runtime services, and factory registries.
Dagger does not define the pipeline DAG.
Flow node ids are resolved only inside a flow.
They are not Dagger binding keys.

=== dsl-kotlin

The `dsl-kotlin` module provides the primary developer API.
It composes `ExecutableFlow` instances in Kotlin.

=== dsl-hocon

The `dsl-hocon` module parses SeaTunnel-style HOCON into `FlowDefinition`.
It does not execute jobs directly.
Execution still goes through `FlowAssembler` and `SparkRuntime`.

=== cli

The `cli` module provides a HOCON file runner.
Use it when users should provide only a configuration file and should not need
to write a Kotlin runner.

== Kotlin DSL

The Kotlin DSL is the native programmatic API for developers.
Use it when pipeline construction should benefit from Kotlin types, functions, branching, and normal application code.
The DSL describes a graph; it does not run Spark actions while the graph is being built.

=== Basic Pipeline

[source,kotlin]
----
import org.apache.spark.sql.SaveMode
import org.openprojectx.spark.boot.dsl.kotlin.SparkBoot
import org.openprojectx.spark.boot.dsl.kotlin.filterSql
import org.openprojectx.spark.boot.dsl.kotlin.runSparkBoot
import org.openprojectx.spark.boot.dsl.kotlin.select
import org.openprojectx.spark.boot.dsl.kotlin.writeParquet

@SparkBoot
fun main(args: Array<String>) = runSparkBoot(args) {
    flow("paid-orders") {
        parquetSource("orders") {
            path = "data/orders"
        }
            .filterSql("paid-only") {
                condition = "status = 'PAID'"
            }
            .select("select-columns") {
                columns = listOf("id", "amount", "status")
            }
            .writeParquet("sink") {
                path = "output/paid-orders"
                mode = SaveMode.Overwrite
            }
    }
}
----

`runSparkBoot` creates the Dagger-backed Spark Boot component, executes the returned flow,
and stops the Spark session in a `finally` block.
The annotation marks the application entry point for discovery and future tooling.

The DSL can live in the current main file or in a separate Kotlin file:

[source,kotlin]
----
fun SparkBootContext.ordersFlow(input: String, output: String) = flow("paid-orders") {
    parquetSource("orders") {
        path = input
    }.writeParquet("sink") {
        path = output
        mode = SaveMode.Overwrite
    }
}

@SparkBoot
fun main(args: Array<String>) = runSparkBoot(args) {
    ordersFlow("data/orders", "output/paid-orders")
}
----

=== User-Provided Dagger Nodes

Applications can contribute node factories through Dagger, similar to how a Spring Boot app contributes beans.
The important difference is that Dagger contributions are compile-time bindings, not runtime bean lookups.

The DSL shape is:

[source,kotlin]
----
node<InMemoryOrdersSourceNode>("orders", "InMemoryOrdersSource") {
    orders = listOf(
        Order(id = "1", amount = 10.0, status = "PAID"),
        Order(id = "2", amount = 20.0, status = "CANCELLED")
    )
}
----

The semantic contract is:

. `"InMemoryOrdersSource"` is the programmatic node kind. It selects a Dagger-registered factory.
. `"orders"` is the flow-local node id. It is used for DAG registration and edge wiring.
. Dagger creates the node through the registered factory.
. The lambda customizes the created node instance.
. The configured node instance is registered into the flow.

Define the node and factory:

[source,kotlin]
----
class InMemoryOrdersSourceNode : SparkSourceNode<Dataset<Row>> {
    var orders: List<Order> = emptyList()

    override val name: String = "in-memory-orders-source"

    override fun execute(input: Unit, context: SparkExecutionContext): Dataset<Row> {
        return context.spark.createDataFrame(orders, Order::class.java)
    }
}

class InMemoryOrdersSourceNodeFactory @Inject constructor() :
    NodeFactory<InMemoryOrdersSourceNode> {
    override fun create(): InMemoryOrdersSourceNode = InMemoryOrdersSourceNode()
}
----

Bind the factory into Spark Boot's programmatic factory map:

[source,kotlin]
----
@Module
interface AppNodesModule {
    @Binds
    @IntoMap
    @StringKey("InMemoryOrdersSource")
    fun bindInMemoryOrdersSourceFactory(
        factory: InMemoryOrdersSourceNodeFactory
    ): UntypedNodeFactory
}
----

Build an app component from Spark Boot modules plus the app module:

[source,kotlin]
----
@Singleton
@Component(
    modules = [
        SparkModule::class,
        RuntimeModule::class,
        BuiltinConnectorModule::class,
        AppNodesModule::class
    ]
)
interface AppComponent : SparkBootComponent
----

Pass that component to `runSparkBoot`:

[source,kotlin]
----
@SparkBoot
fun main(args: Array<String>) = runSparkBoot(args, DaggerAppComponent.create()) {
    flow("paid-orders") {
        node<InMemoryOrdersSourceNode>("orders", "InMemoryOrdersSource") {
            orders = listOf(Order(id = "1", amount = 10.0, status = "PAID"))
        }
            .filterSql("paid-only") {
                condition = "status = 'PAID'"
            }
            .writeParquet("sink") {
                path = "output/paid-orders"
                mode = SaveMode.Overwrite
            }
    }
}
----

=== Starter-Style Configuration

Spark Boot has a small autoconfigure layer for shared infrastructure settings.
It is intentionally similar to Spring Boot configuration: define infrastructure
once in environment config, then refer to it by logical name from HOCON or the
Kotlin DSL.

Spark Boot reads `spark.boot` settings from Typesafe Config, so classpath
`application.conf`, system properties, and environment-backed config sources can
contribute values before the Dagger component creates the Spark session.
Partial config is supported: stable values can live in
`src/main/resources/application.conf`, and deployment-specific or test-specific
values can be supplied through system properties.

Profiles are supported for classpath config. Spark Boot loads
`application.conf` first, then overlays `application-<profile>.conf` for each
active profile in declaration order. System properties remain the final override
layer.

[source,hocon]
----
spark.boot.profiles.active = "local,ci"
----

Profiles can also be supplied at launch:

[source,bash]
----
java -Dspark.boot.profiles.active=ci ...
spark-boot-cli --profile ci job.conf
----

[source,hocon]
----
spark.boot {
  s3 {
    endpoint = "http://localhost:4566"
    region = "us-east-1"
    access-key = "test"
    secret-key = "test"
    path-style-access = true
    ssl-enabled = false
  }

  hms {
    uri = "thrift://localhost:9083"
    warehouse = "s3a://warehouse/iceberg"
    catalog = "hms"
  }

  jdbc.connections.orders {
    url = "jdbc:mysql://localhost:3306/orders"
    user = "spark"
    password = "spark"
    driver = "com.mysql.cj.jdbc.Driver"
  }
}
----

For example, an application can keep this in
`src/main/resources/application.conf`:

[source,hocon]
----
spark.boot {
  hms {
    catalog = "hms"
  }

  jdbc.connections.orders {
    user = "spark"
    password = "spark"
    driver = "com.mysql.cj.jdbc.Driver"
  }
}
----

Then a launcher or test can provide only the dynamic pieces before creating the
Dagger component:

[source,kotlin]
----
System.setProperty("spark.boot.jdbc.connections.orders.url", jdbcUrl)
System.setProperty("spark.boot.hms.uri", metastoreUri)
System.setProperty("spark.boot.hms.warehouse", warehouse)
----

The contract is:

. `spark.boot.s3` configures Hadoop `s3a` before `SparkSession.getOrCreate()`.
. `spark.boot.hms` enables Hive support and registers the named Iceberg catalog.
. `spark.boot.jdbc.connections.<name>` creates a named JDBC connection.
. A node may still provide explicit `url`, `user`, `password`, or `driver`; explicit node values override the named connection values.
. Dynamic tests may set only the missing paths as system properties before creating the Dagger component, for example `spark.boot.jdbc.connections.orders.url`.
. Profile config files are ordinary Typesafe Config resources named `application-<profile>.conf`.
. If you create a custom Dagger component yourself, activate CLI profiles before calling `Dagger...Component.create()`.

Kotlin DSL:

[source,kotlin]
----
jdbcSource("orders") {
    connection = "orders"
    table = "jdbc_orders"
}.writeIceberg("sink") {
    catalog = "hms"
    table = "default.jdbc_orders"
    mode = SaveMode.Overwrite
}
----

HOCON flow:

[source,hocon]
----
source = [
  {
    plugin_name = "Jdbc"
    plugin_output = "orders"
    connection = "orders"
    table = "jdbc_orders"
  }
]

sink = [
  {
    plugin_name = "Iceberg"
    plugin_input = "orders"
    catalog = "hms"
    table = "default.jdbc_orders"
    save_mode = "overwrite"
  }
]
----

=== Explicit Edges

For branching or non-linear flows, assign node references and connect them explicitly.

[source,kotlin]
----
val flow = sparkFlow("orders", component) {
    val orders = parquetSource("orders") {
        path = "data/orders"
    }

    val paid = orders.filterSql("paid") {
        condition = "status = 'PAID'"
    }

    val cancelled = orders.filterSql("cancelled") {
        condition = "status = 'CANCELLED'"
    }

    paid.writeParquet("paid-sink") {
        path = "output/paid"
        mode = SaveMode.Overwrite
    }

    cancelled.writeParquet("cancelled-sink") {
        path = "output/cancelled"
        mode = SaveMode.Overwrite
    }
}
----

This creates two branches from the `orders` source.
The current runtime supports fan-out.
It does not support multi-input joins yet.

=== Spark-Native Escape Hatch

Spark classes are intentionally absent from the core model.
They are exposed only through explicit Spark-native DSL APIs.

[source,kotlin]
----
val flow = sparkFlow("advanced-orders", component) {
    parquetSource("orders") {
        path = "data/orders"
    }
        .transformDataFrame("add-derived-amount") { df ->
            df.withColumn(
                "amount2",
                org.apache.spark.sql.functions.col("amount").multiply(2)
            )
        }
        .writeParquet("sink") {
            path = "output/orders2"
            mode = SaveMode.Overwrite
        }
}
----

Use explicit names such as `transformDataFrame` because the code runs under Spark execution semantics.

=== Profiled Dagger Contributions

Dagger does not dynamically enable or disable modules at runtime the way Spring
does. Spark Boot supports the Spring-style bean experience at the registry
boundary instead: a Dagger module can contribute profile-specific node factories,
and Spark Boot exposes only the factories whose profiles are active.

[source,kotlin]
----
@Module
interface AppModule {
    companion object {
        @Provides
        @IntoSet
        fun localOrders(factory: LocalOrdersSourceFactory) =
            ProfiledProgrammaticNodeFactory(
                kind = "OrdersSource",
                profiles = setOf("local"),
                factory = factory
            )

        @Provides
        @IntoSet
        fun ciOrders(factory: CiOrdersSourceFactory) =
            ProfiledProgrammaticNodeFactory(
                kind = "OrdersSource",
                profiles = setOf("ci"),
                factory = factory
            )
    }
}
----

The DSL keeps the same shape for every environment:

[source,kotlin]
----
node<OrdersSourceNode>("orders", "OrdersSource") {
}
----

With profile `local`, Spark Boot creates `OrdersSource` through
`LocalOrdersSourceFactory`. With profile `ci`, it uses `CiOrdersSourceFactory`.
If multiple active profiled factories contribute the same kind, Spark Boot fails
fast because the binding would be ambiguous.

The same pattern is available for config-driven HOCON nodes through
`ProfiledConfigNodeFactory`.

== HOCON DSL

The HOCON DSL is a SeaTunnel-style declarative API for runtime and deployment configuration.
It should be used for portable jobs that do not need arbitrary Kotlin or JVM code.

Supported top-level blocks:

* `env`
* `source`
* `transform`
* `sink`

=== Canonical Array Style

[source,hocon]
----
env {
  job.name = "paid-orders"
  job.mode = "BATCH"
  parallelism = 4
}

source = [
  {
    plugin_name = "Parquet"
    path = "data/orders"
    plugin_output = "orders"
  }
]

transform = [
  {
    plugin_name = "Sql"
    plugin_input = "orders"
    plugin_output = "paid_orders"
    query = "select id, amount, status from orders where status = 'PAID'"
  }
]

sink = [
  {
    plugin_name = "Parquet"
    plugin_input = "paid_orders"
    path = "output/paid-orders"
    save_mode = "overwrite"
  }
]
----

=== Object Style

[source,hocon]
----
env {
  job.name = "paid-orders"
  job.mode = "BATCH"
}

source {
  Parquet {
    path = "data/orders"
    plugin_output = "orders"
  }
}

transform {
  Sql {
    plugin_input = "orders"
    plugin_output = "paid_orders"
    query = "select id, amount, status from orders where status = 'PAID'"
  }
}

sink {
  Parquet {
    plugin_input = "paid_orders"
    path = "output/paid-orders"
    save_mode = "overwrite"
  }
}
----

=== Compatibility Aliases

The parser accepts these output aliases:

* `source_table_name` maps to `plugin_output` for sources
* `result_table_name` maps to `plugin_output` for transforms

Prefer `plugin_output` for new configuration.

=== Mapping Rules

Source item:

[source,text]
----
node id = plugin_output
node type = <plugin_name>Source
node config = item config
----

Transform item:

[source,text]
----
node id = plugin_output
node type = <plugin_name>Transform
edge = plugin_input -> plugin_output
node config = item config
----

Sink item:

[source,text]
----
node id = sink_<plugin_input>_<index>
node type = <plugin_name>Sink
edge = plugin_input -> sink node id
node config = item config
----

=== Running HOCON

The parser returns a `FlowDefinition`.
Assemble it with Dagger-backed factories before execution.

[source,kotlin]
----
import com.typesafe.config.ConfigFactory
import java.io.File
import org.openprojectx.spark.boot.core.FlowAssembler
import org.openprojectx.spark.boot.dagger.DaggerSparkBootComponent
import org.openprojectx.spark.boot.dsl.hocon.SeaTunnelStyleConfigParser

val component = DaggerSparkBootComponent.create()
val config = ConfigFactory.parseFile(File("job.conf")).resolve()
val definition = SeaTunnelStyleConfigParser().parse(config)
val executableFlow = FlowAssembler(component.nodeFactoryRegistry()).assemble(definition)

component.sparkRuntime().run(executableFlow)
----

The `cli` module packages that runner flow:

[source,bash]
----
java -cp "<app-and-dependencies>" org.openprojectx.spark.boot.cli.SparkBootCliKt paid-orders.conf
----

The CLI expects one argument: the HOCON config file path.

== Built-In Nodes

=== ParquetSource

HOCON type: `ParquetSource`

Required config:

* `path`

Kotlin DSL:

[source,kotlin]
----
parquetSource("orders") {
    path = "data/orders"
}
----

=== JdbcSource

HOCON type: `JdbcSource`

Required config:

* `table`

Optional config:

* `connection`
* `url`
* `user`
* `password`
* `driver`

If `connection` is set, Spark Boot resolves `url`, `user`, `password`, and
`driver` from `spark.boot.jdbc.connections.<connection>`.
Explicit node values override values from the named connection.

Kotlin DSL:

[source,kotlin]
----
jdbcSource("orders") {
    connection = "orders"
    table = "orders"
}
----

* `save_mode` or `mode`

If `catalog` is set and `table` is not already prefixed with that catalog,
Spark Boot writes to `<catalog>.<table>`.

For HMS-backed Iceberg, prefer `spark.boot.hms` config before the Spark session
is created. The legacy compatibility settings still work:

* `hive.metastore.uris` system property or `HIVE_METASTORE_URIS` environment variable
* `spark.boot.iceberg.warehouse` system property or `SPARK_BOOT_ICEBERG_WAREHOUSE` environment variable

When HMS is configured, Spark Boot enables Hive support and registers the `hms` Iceberg catalog.
Use table names such as `hms.default.paid_orders`.

== Runtime Validation

`SparkRuntime` validates the executable DAG before running it.

Validation checks:

* the flow has at least one node
* all edge endpoints exist
* the graph has at least one source and one sink
* the graph has no cycles
* each node has at most one input
* topological ordering is possible

Input handling:

* source nodes receive `Unit`
* transform nodes receive the upstream node output
* sink nodes receive the upstream node output
* multi-input nodes fail in this version

== Extending Spark Boot

=== Add a Programmatic Node

Create a node that implements one of the Spark node interfaces:

[source,kotlin]
----
class MyTransformNode : SparkTransformNode<Dataset<Row>, Dataset<Row>> {
    override val name: String = "my-transform"

    override fun execute(
        input: Dataset<Row>,
        context: SparkExecutionContext
    ): Dataset<Row> {
        return input
    }
}
----

Add a programmatic factory:

[source,kotlin]
----
class MyTransformNodeFactory @Inject constructor() : NodeFactory<MyTransformNode> {
    override fun create(): MyTransformNode = MyTransformNode()
}
----

Bind it into the programmatic factory map:

[source,kotlin]
----
@Binds
@IntoMap
@StringKey("MyTransform")
fun bindMyTransformFactory(factory: MyTransformNodeFactory): UntypedNodeFactory
----

Then Kotlin DSL can create it by kind and register it by flow-local id:

[source,kotlin]
----
node<MyTransformNode>("normalize-orders", "MyTransform") {
    // configure node properties here
}
----

=== Add a HOCON Node

Create a config factory:

[source,kotlin]
----
class MyTransformConfigFactory @Inject constructor() : ConfigNodeFactory {
    override fun create(config: Map<String, Any?>): FlowNode<*, *> {
        return MyTransformNode()
    }
}
----

Bind it into the Dagger map:

[source,kotlin]
----
@Binds
@IntoMap
@StringKey("MyTransform")
fun bindMyTransformFactory(
    factory: MyTransformConfigFactory
): ConfigNodeFactory
----

Then HOCON can refer to:

[source,hocon]
----
transform = [
  {
    plugin_name = "My"
    plugin_input = "orders"
    plugin_output = "orders2"
  }
]
----

The parser maps that to type `MyTransform`.

== Build And Test

Run the full test suite:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew test --no-configuration-cache
----

Run only integration tests:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :integration-tests:test --no-configuration-cache
----

Publish the root project to Maven local before running examples:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew publishToMavenLocal --no-configuration-cache
----

Run the standalone examples build:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew -p examples runAll --no-configuration-cache
env GRADLE_USER_HOME=/data/.gradle ./gradlew -p examples :kotlin-dsl:run --no-configuration-cache
env GRADLE_USER_HOME=/data/.gradle ./gradlew -p examples :spark-boot-app:run --no-configuration-cache
env GRADLE_USER_HOME=/data/.gradle ./gradlew -p examples :hocon:run --no-configuration-cache
env GRADLE_USER_HOME=/data/.gradle ./gradlew -p examples :jdbc-iceberg-hms:run --no-configuration-cache
----

The `examples` directory is an independent multi-module Gradle build.
It is excluded from the root project and consumes `org.openprojectx.spark.boot:*:0.1.0-SNAPSHOT` artifacts from Maven local.
The Kotlin DSL examples create temporary Parquet input in code.
The `:spark-boot-app` module demonstrates the `@SparkBoot` application entry point and a user-provided Dagger node factory as a runnable application.
The HOCON example is config-only: `org.openprojectx.bigdata-test` starts LocalStack S3 and prepares the Parquet input from TOML before the Spark Boot CLI runs `paid-orders.conf`.
The `:jdbc-iceberg-hms` module is both a runnable app and a JUnit-backed integration example.
It starts LocalStack S3 and Hive Metastore through `org.openprojectx.bigdata-test`; the app itself starts a MariaDB Testcontainers source and runs equivalent HOCON and Kotlin DSL flows into HMS-backed Iceberg tables.

== Current Limitations

Version 1 intentionally does not implement:

* full Apache SeaTunnel runtime compatibility
* dynamic runtime plugin loading
* distributed scheduling
* cluster submission management
* streaming checkpoint lifecycle management
* arbitrary scripting inside HOCON
* multi-input joins
* automatic schema inference framework
* visual DAG editing
